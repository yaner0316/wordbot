# 迭代工作流

## 本轮有效工作顺序

1. 先确认根因，不猜
2. 先写或修改测试，再改实现
3. 先跑后端单测
4. 再跑前端结构测试
5. 最后再看浏览器体验

## 适用场景

这个顺序适用于：

- 修改掌握规则
- 调整 UI
- 做性能优化
- 改后端接口
- 扩展家长管理能力
- 调整复习机制

## 协作原则

- 不直接凭感觉改规则
- 先把预期行为写进测试或文档
- 改动后必须跑对应回归
- 性能问题先加观测，再做优化
- 数据迁移类工作单独拆分，避免和业务规则修改混在一起
