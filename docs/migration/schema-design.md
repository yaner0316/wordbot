# WordBot Postgres/Supabase Schema Design (Migration Gate 1)

## Scope

This design is the end-state relational model after Feishu is fully replaced. It is based on `docs/migration/feishu-inventory.md` and current semantics in `backend/feishu.js`, `backend/quiz-word-queue.js`, `backend/mastery-evidence.js`, `backend/auth-service.js`, and `backend/question-cache.js`.

This gate changes no runtime code, performs no Feishu writes, connects to no database, and defines no backfill implementation. `schema.sql` is first-run DDL for a new Supabase Postgres database.

## Recommended Model

| Table | Purpose |
| --- | --- |
| `users` | One canonical student identity plus child/parent credentials, phone, and user-level learning settings. |
| `words` | User-owned vocabulary/meaning rows. Multiple rows for the same spelling are allowed because mastery is meaning-row based. |
| `parts_of_speech` | Extensible normalized POS vocabulary. |
| `word_parts_of_speech` | Ordered junction preserving compound POS such as `noun, adjective`. |
| `assessments` | Generated and submitted quiz-question history, including immutable question/word snapshots and normalized submission result. |
| `question_cache` | Stable cache variants linked to a word row; duplicate natural-key groups are intentionally preserved. |

No permanent Feishu mirror tables are proposed. Migration-only raw extracts or quarantine tables should live outside this end-state schema.

## Primary Keys and Feishu Reconciliation

All mutable business tables use UUID surrogate primary keys generated by `gen_random_uuid()`.

Reasons:

1. New post-cutover records will not have Feishu IDs.
2. Foreign keys should not depend on an external vendor identifier.
3. UUIDs are directly supported by Postgres/Supabase and safe for distributed creation.
4. Feishu record IDs remain useful for reconciliation, but not as the permanent identity contract.

`users`, `words`, `assessments`, and `question_cache` each include a nullable, unique `feishu_record_id`. It is nullable because records created after cutover have no Feishu source. For `users`, this is the WORD record that supplied migrated account/auth fields; Feishu did not have a separate users table.

The TEST field named `record_id` and QUESTION_CACHE field named `word_record_id` are references to a WORD record, not the source row's own ID. They become `word_id` foreign keys. Their exact source text can also be retained in `source_word_record_id` for reconciliation.

Natural keys are not primary keys:

- `(user, word)` has known duplicates in WORD.
- TEST has duplicate `(user, test_id, record_id)` groups.
- QUESTION_CACHE has 126 duplicate natural-key groups whose content differs and must not be deduplicated.

## Canonical User Identity

The canonical lookup rule matches `backend/auth-service.js`:

1. Trim leading/trailing whitespace.
2. Remove all remaining whitespace characters.
3. Lowercase the result to form `username_key`.
4. Preserve the chosen product-facing casing in `username`.

Examples:

- `" Draggy "` -> key `draggy`, display `Draggy`
- `"draggy"` -> key `draggy`, merged into the same user

`users.username_key` is a stored generated column with a unique constraint. Every word, assessment, and cache row references `users.id`, never a username string. Backfill resolves all raw variants to the canonical user before inserting child rows.

`test_user1` is not imported because it is an empty shell. `test_user` remains a normal canonical user and its 50 words are retained.

The parent username uses the same whitespace/case key for login comparison, but is not globally unique because parent identity is scoped to a child account in the current product.

## Users and Auth Fields

The auth fields currently embedded in WORD rows move to `users`:

- `auth_password_hash` -> `password_hash`
- `auth_password_salt` -> `password_salt`
- `auth_created_at` -> `auth_created_at`
- `parent_username` -> `parent_username`
- `parent_password_hash` -> `parent_password_hash`
- `parent_password_salt` -> `parent_password_salt`
- `parent_created_at` -> `parent_created_at`
- `phone` -> `phone`
- `phone_verified_at` -> `phone_verified_at`

Credential checks require hash/salt pairs to be both null or both populated. Parent credentials also require a nonblank parent username. Password values remain nullable so users without initialized credentials can be represented.

`Learning_Level` and `Level_Changed_At` also move to `users` because they are account-level learning settings, not properties of an individual word.

## Word Semantics

`words` stores one user-owned meaning row. It intentionally does not enforce uniqueness on `(user_id, lower(word))`, because the inventory contains duplicate spellings with distinct meanings and `mastery-evidence.js` evaluates each meaning record independently before declaring the spelling mastered.

`mastery_status` supports the evidence stages `pending`, `recognized`, `consolidating`, and `mastered`. Backfill mappings are:

- `Pending` and `optXjbXS2F` -> `pending`
- `Mastered` and `optF5P0W3O` -> `mastered`

`remember_time` becomes `remembered_at`. `record_time` becomes `entered_at` and supplies FIFO order.

`Distractors` and `Old_Distractors` become JSON arrays because they are ordered question-building data. `Quality_Flags` becomes `text[]`; `Quality_Note` remains text. Feishu `Question_Cache_Status` is omitted because cache state is normalized in `question_cache` and can be derived per user/word.

### Part of Speech

POS is not a Postgres enum. It is extensible language metadata and already contains compound values, so a lookup plus ordered junction is safer:

- Normalize abbreviations such as `n.` -> `noun`, `v.` -> `verb`, `adj.` -> `adjective`, and `adv.` -> `adverb` during backfill.
- Split a compound source such as `noun, adjective` into two ordered junction rows.
- Keep multiword values such as `phrasal verb` and `verb phrase` as single lookup values.

This preserves compound POS without locking future additions behind an enum migration.

### Multiple Definitions

`multi_definition` is a three-state enum:

- `opthB7bmkB` -> `yes`
- `optpWwFJpq` -> `no`
- `optH7bmkB` -> `unknown`

The five unknown rows retain `source_multi_definition_option_id = 'optH7bmkB'`. They are deliberately flagged rather than silently treated as yes or no.

## Assessment Semantics

Each assessment row is one generated question/attempt row. `test_id` remains the grouping key used by history screens. `assessed_at` comes from `test_time`.

`word_id` links the attempt to a meaning row when that row still exists. It uses `ON DELETE SET NULL`, not cascade, because current word deletion preserves TEST history. `word_snapshot` and question fields preserve historical display after a word is deleted.

`is_correct` is a nullable enum:

- `optHGT7gYf` -> `correct`
- `optbe4bsQk` -> `wrong`
- null -> generated but not submitted

The submitted answer is split into `submitted_answer` and `answer_confidence`. Legacy scored rows without explicit confidence backfill to `sure`, matching `mastery-evidence.js`. A scored row requires both fields, while an answer may remain present with null correctness because the inventory has more `your_answer` values than `is_correct` values.

`learning_day` is stored explicitly as the Asia/Shanghai calendar date derived from `assessed_at`. This avoids repeating timezone expressions in the same-day de-duplication query and makes the product's learning-day boundary auditable.

`is_real_assessment` normalizes the current convention (`test_id = 'real'` or starts with `real-`). Mastery and same-day indexes exclude test/demo rows without repeatedly parsing `test_id`.

Review fields remain nullable text because the inventory has too little value diversity to infer safe closed domains for `assessment_kind`, `review_round`, and `review_status`.

## Question Cache Semantics

Each cache row has its own UUID and optional unique Feishu ID. There is intentionally no unique constraint on `(user_id, word_id, question_type, round_type, level)`. All 126 duplicate natural-key groups are imported as separate variants because their content differs.

The cache links to both `users` and `words`. A composite foreign key `(word_id, user_id)` ensures the cached word belongs to the same user. Deleting a word cascades to its cache rows, matching current cleanup behavior.

`options` and `option_meanings` are JSON arrays because ordering is meaningful and cache readiness expects exactly four ordered values. `answer` is constrained to A/B/C/D. `used_count` is a nonnegative bigint, allowing safe conversion from its current string representation.

## Enum Recommendation

Postgres enums are recommended for small, closed, application-controlled domains that participate directly in constraints and partial indexes:

| Domain | Values |
| --- | --- |
| `wordbot_level` | `小学`, `中学`, `高中`, `CET4_6_TOEFL` |
| `question_type` | `1`, `2`, `3`, `4` |
| `round_type` | `primary`, `review` |
| `question_quality_status` | `pending`, `ready`, `failed`, `stale` |
| `correctness_status` | `correct`, `wrong` |
| `mastery_status` | `pending`, `recognized`, `consolidating`, `mastered` |

Additional closed domains use enums for `multi_definition_status`, `answer_confidence`, and `question_source`.

Enums prevent misspelled states and make the state machine visible. Removing or renaming enum values requires a migration. POS uses lookup tables because it is extensible and compound.

## Foreign Keys and Delete Behavior

| Relationship | Delete behavior | Reason |
| --- | --- | --- |
| `words.user_id -> users.id` | `CASCADE` | Deleting a user removes owned learning data. |
| `assessments.user_id -> users.id` | `CASCADE` | Account deletion removes personal assessment history. |
| `assessments.word_id -> words.id` | `SET NULL` | Preserve historical attempts and snapshots after word deletion. |
| `question_cache.user_id -> users.id` | `CASCADE` | Cache is user-owned and disposable. |
| `question_cache.(word_id,user_id) -> words.(id,user_id)` | `CASCADE` | Enforce same-user ownership and remove stale cache on word deletion. |
| `word_parts_of_speech.word_id -> words.id` | `CASCADE` | POS assignments have no life outside the word row. |
| `word_parts_of_speech.part_of_speech_id -> parts_of_speech.id` | `RESTRICT` | Prevent deletion of a POS value still in use. |

## Indexes and Hot Paths

### Login

- `users_username_key_unique` supplies canonical username lookup and prevents casing/whitespace duplicates.

### Word-Queue Quiz Generation

- `words_quiz_queue_idx (user_id, level, entered_at, id) WHERE mastery_status <> 'mastered'` serves user + level filtering and stable FIFO ordering.
- `assessments_same_day_dedup_idx (user_id, learning_day, word_id) WHERE is_real_assessment AND is_correct IS NOT NULL` serves same-day exclusion for submitted real attempts.
- `assessments_mastery_evidence_idx (user_id, word_id, assessed_at) INCLUDE (...) WHERE is_real_assessment AND is_correct IS NOT NULL` serves chronological mastery evidence by meaning row.
- `words_user_word_idx (user_id, lower(word))` serves grouping same-spelling meaning rows without false uniqueness.

The expected queue plan is: obtain eligible FIFO word rows, anti-join today's assessment word IDs, then consult ordered evidence for non-materialized mastery checks or compare maintained `mastery_status`.

### Cache Selection

- `question_cache_ready_pool_idx (user_id, level, round_type, used_count, generated_at, id) ... WHERE quality_status = 'ready'` serves the ready pool and least-used/oldest frontier.
- `question_cache_ready_word_idx (user_id, word_id, level, round_type, used_count, generated_at, id) WHERE quality_status = 'ready'` joins the selected word queue to ready variants while preserving duplicate content rows.

### History and Reconciliation

- `assessments_user_history_idx (user_id, assessed_at DESC, test_id)` serves history grouping/sorting.
- `assessments_source_word_record_idx (source_word_record_id)` and `question_cache_source_word_record_idx (source_word_record_id)` support reconciliation by original Feishu WORD record ID.
- Each `feishu_record_id UNIQUE` constraint creates its own reconciliation index.

### Complete Index Catalog

| Index or constraint-backed index | Query/constraint served |
| --- | --- |
| `users_username_key_unique` | Canonical login lookup and duplicate-identity prevention. |
| Per-table `feishu_record_id UNIQUE` indexes | Exact source-row reconciliation for users, words, assessments, and cache. |
| `words_quiz_queue_idx` | User/level/not-mastered FIFO queue scan. |
| `words_user_word_idx` | Same-spelling meaning grouping and administrative word lookup. |
| `assessments_same_day_dedup_idx` | Today's submitted real word IDs for queue anti-join. |
| `assessments_mastery_evidence_idx` | Ordered submitted real evidence for mastery calculation. |
| `assessments_user_history_idx` | Newest-first user assessment history grouped by `test_id`. |
| `question_cache_ready_pool_idx` | Ready cache frontier by user, level, round, usage, and age. |
| `question_cache_ready_word_idx` | Ready cache variants for word IDs selected by the queue. |
| `assessments_source_word_record_idx` | Backfill and audit lookup by legacy TEST `record_id`. |
| `question_cache_source_word_record_idx` | Backfill and audit lookup by legacy CACHE `word_record_id`. |
| Primary-key indexes | Direct row lookup and all foreign-key targets. |
| `words_id_user_unique` | Composite cache ownership foreign-key target. |
| POS unique/primary-key indexes | POS code uniqueness, junction identity, and stable compound ordering. |

## Timestamp and Numeric Conversion

Feishu epoch-millisecond values convert during backfill with the equivalent of:

```sql
to_timestamp((source_value::numeric) / 1000.0)
```

| Feishu field | Postgres field | Type |
| --- | --- | --- |
| WORD `record_time` | `words.entered_at` | `timestamptz` |
| WORD `remember_time` | `words.remembered_at` | `timestamptz` |
| WORD `auth_created_at` | `users.auth_created_at` | `timestamptz` |
| WORD `parent_created_at` | `users.parent_created_at` | `timestamptz` |
| WORD `phone_verified_at` | `users.phone_verified_at` | `timestamptz` |
| WORD `Level_Changed_At` | `users.level_changed_at` | `timestamptz` |
| TEST `test_time` | `assessments.assessed_at` | `timestamptz` |
| CACHE `generated_at` | `question_cache.generated_at` | `timestamptz` |
| CACHE `last_used_at` | `question_cache.last_used_at` | `timestamptz` |
| CACHE `used_count` | `question_cache.used_count` | `bigint` |
| WORD `Error_Count` | `words.error_count` | `integer` |

Although CACHE timestamps and counters are strings today, only validated numeric strings should be cast. Blank strings become null for nullable timestamps. `learning_day` is computed from `assessed_at` in `Asia/Shanghai` during backfill and on every new insert.

## Locked Cleanup Rules

- Do not import the 10 fully empty TEST rows.
- Do not import the `test_user1` empty shell.
- Merge `draggy` into canonical display user `Draggy` using key `draggy`.
- Keep `test_user` and all 50 of its words.
- Map the owner-identified corrupted elementary level (`Сѧ`) and `??` to `小学`; normalize other known mojibake labels only when the mapping is deterministic.
- Normalize POS abbreviations and preserve compound POS through the junction table.
- Map known Feishu option IDs to the human enum values listed above.
- Mark `optH7bmkB` as `unknown` and retain the raw option ID.
- Do not deduplicate question-cache natural-key collisions.

## Supabase Security Posture

The DDL enables Row Level Security and creates no client policies. This is deny-by-default for Supabase Data API roles. A protected server-side service role can perform later migration/runtime work, while a later auth/API gate must define least-privilege policies before browser clients receive direct table access.

This design preserves existing PBKDF2 hash/salt fields and keeps authentication against `public.users`. It does not link `users` to `auth.users` for Gate 1. Supabase Auth remains a possible future rolling migration only, using progressive re-hash on successful login if a later product/security gate approves it.

## Resolved Migration Decisions

1. **RESOLVED - Three malformed WORD rows:** The rows missing `Word`, `Meaning`, and `Status` are treated as account-carrier rows. Backfill imports their auth/account fields into `users` only and must not create `words` rows from them. `words` intentionally rejects them as vocabulary rows. Gate 2 must verify row-level auth/account fields for `recvnacJpRa48s`, `recvnfE765MZdP`, and `recvpw4lpJj2uE`; if any of the three is not an account row, the migration must stop and flag it rather than silently dropping real vocabulary.
2. **RESOLVED - 135 WORD rows missing `Level`:** Backfill inherits `users.learning_level`. If the user also has no `learning_level`, leave `words.level` null and exclude the word from level-specific queues until repaired. The DDL keeps `words.level` nullable rather than guessing from cache/history.
3. **RESOLVED - One TEST row missing `correct_answer`:** Import it with `correct_answer` null and a migration flag noting the missing answer. This preserves legacy fidelity and avoids quarantine for a single historical gap. The DDL already permits null `correct_answer`.
4. **RESOLVED - 48 TEST answer/result mismatches:** The inventory has 692 `your_answer` values but only 644 `is_correct` values. Import answer-present/result-null rows as-is with `submitted_answer` populated, `is_correct` null, and a migration flag such as `answer-present-result-null`. They remain pending; backfill must not invent correctness.
5. **RESOLVED - Supabase Auth:** Keep custom PBKDF2 authentication against `public.users`. Existing hashes migrate as-is, with zero password resets and unchanged login UX. Supabase Auth is deferred to a possible future rolling migration, likely progressive re-hash on successful login, not part of Gate 1.
6. **RESOLVED - Phone uniqueness:** Do not add a unique constraint on `phone` for now. Product has not confirmed one-phone-one-child semantics, so phone remains plain nullable text.
7. **RESOLVED - Assessment retention on word deletion:** Preserve assessment history when a word is deleted. The current `assessments.word_id ON DELETE SET NULL` behavior stands; cache rows still cascade because cache is disposable.
8. **RESOLVED - Enum evolution:** Keep enums for closed, application-controlled level and question-type domains. New levels or question types must go through reviewed schema migrations. Revisit lookup tables only if administrators need to add these values dynamically at runtime.

## Backfill Validation Expectations for Gate 2

The later backfill should fail or quarantine, never silently coerce, when a source option ID, timestamp, counter, JSON list, or foreign record ID is unknown. It should reconcile source counts after locked deletions, prove every child row resolves to one canonical user, prove every cache row resolves to a word owned by that user, and report every resolved nullable case separately, including account-carrier rows, inherited or null word levels, null legacy correct answers, and answer-present/result-null assessments.
